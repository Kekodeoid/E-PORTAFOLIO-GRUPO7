Contexto del Proyecto

Introducción y Problemática

La brecha digital en el sector salud privado hondureño representa una oportunidad concreta de impacto operativo y económico.

🏥 El Problema
La mayoría de clínicas privadas de mediano tamaño en Honduras aún dependen de llamadas telefónicas, agendas en papel u hojas de cálculo para manejar sus citas médicas. Esto genera altas tasas de inasistencia (25–35%), errores en registro, sobrecarga de recepción y expedientes que pueden perderse o dañarse.
💡 La Oportunidad
WhatsApp tiene una penetración superior al 90% entre usuarios de smartphone en Honduras, convirtiéndolo en el canal ideal para comunicación automatizada con pacientes. Esto, combinado con una plataforma web de gestión, puede transformar la operación de una clínica privada de forma medible y sostenible.
🎯 Nuestra Propuesta
Desarrollo de un Software de Automatización de Citas Médicas: solución web que integra gestión de agendas multi-médico, envío de recordatorios automáticos vía WhatsApp Business API, y un módulo de expedientes clínicos digitales, todo en 12 semanas con un equipo de 5 personas.
📊 Impacto Esperado
Reducir la tasa de inasistencia del 30% al ≤ 12%, ahorrar 2–3 horas/día en recepción, incrementar un 15–20% las citas efectivas por semana y recuperar la inversión en menos de 7 meses de operación.

Alcance del Proyecto

Declaración de Alcance

Diseñar, desarrollar, probar e implementar un sistema web para la automatización de citas médicas en clínicas privadas en 12 semanas calendario.

✅ Lo que SÍ incluye — v1.0
  • Módulo de gestión de citas y calendario multi-médico
  • Integración WhatsApp Business API — recordatorios y confirmaciones automáticas
  • Expediente clínico digital — historial, diagnósticos, recetas
  • Panel administrativo con métricas de asistencia y ocupación
  • Portal web para pacientes — consulta de citas e historial
  • Sistema de roles y permisos — Administrador, Médico, Recepcionista, Paciente
  • Reportes exportables en PDF y Excel
  • API REST documentada para futuras integraciones
  • Manual de usuario y documentación técnica
  • Capacitación inicial al personal clave de la clínica
🚫 Lo que NO incluye — v1.0
  • Integración con sistemas de laboratorio externos
  • Aplicación móvil nativa (iOS/Android) — el portal web es responsivo
  • Funcionalidades de telemedicina o video consulta
  • Gestión de inventario de medicamentos o insumos
  • Integración con sistemas de seguros médicos nacionales
  • Soporte para múltiples sucursales (una sola sede en v1.0)
  • Migración de expedientes en papel de historial previo al sistema
⚠️ Supuestos Clave
Conectividad ≥ 10 Mbps · Acceso a API WhatsApp Business · 5 integrantes con 20 hrs/semana · Duración máxima 12 semanas · Presupuesto fijo sin incremento

Plan de Calidad y Control de Versiones

Aseguramiento y Control de Calidad

Basado en modelo ISO/IEC 25010. Calidad no es una revisión final, sino un hábito desde el primer día del proyecto.

🎯 Objetivos de Calidad
  • Seguridad de datos clínicos: Expedientes cifrados AES-256, acceso restringido por roles (RBAC).
  • Velocidad y disponibilidad: Respuesta < 2 segundos. Uptime 99.5% en horario de atención.
  • Reducción de no-shows: De 25–35% actual a ≤ 12% en los primeros dos meses de uso.

Jest (Backend) Testing Library (Frontend) Cypress (E2E) OWASP ZAP (Seguridad) SonarCloud (Análisis Código) JMeter (Carga) Postman (API) GitHub Actions (CI/CD) AWS CloudWatch (Monitoreo)
✓ Todas las herramientas QA son Open Source — Costo L 0.00 en Fase 4
🛡️ Aseguramiento de Calidad (Preventivo)
  • Revisión de código entre pares antes de cada merge a rama principal
  • Análisis estático automático con SonarCloud en pipeline GitHub Actions
  • Reuniones semanales los viernes para revisar avance y desviaciones
  • Checklist de diseño aprobado antes de iniciar cualquier módulo
  • Documentación de endpoints en Swagger durante el desarrollo
✅ Control de Calidad (Detectivo — Fase 4)

Cuestionario de Calidad

Reflexión y Análisis del Equipo

Respuestas fundamentadas del equipo a preguntas clave sobre gestión de calidad y control de configuración en sistemas de salud.

Pregunta 1: ¿Por qué es vital el control de configuración en software de salud?
En salud, la precisión no es opcional. Un cambio que parece menor puede tener consecuencias reales: si un desarrollador ajusta el texto del recordatorio de WhatsApp sin avisar y rompe el enlace de confirmación, al día siguiente la recepcionista se encuentra con veinte citas sin confirmar. Un parche no probado en los roles puede dejar que una recepcionista acceda a historiales que no le corresponden. Además, en cualquier auditoría o situación legal, el sistema debe poder demostrar quién cambió qué y cuándo. GitHub provee el historial del código, y el registro de cambios en el SRS cubre las decisiones sobre requisitos, formando juntos una cadena documental defendible.
Pregunta 2: Diferencia entre Aseguramiento y Control de Calidad en este proyecto
El aseguramiento trabaja sobre el proceso, antes de construir, para prevenir problemas. El control trabaja sobre el producto ya construido, para detectar lo que falló. En nuestro proyecto aseguramos calidad cuando definimos que ningún pull request entra sin revisión, cuando configuramos SonarCloud en el pipeline y cuando exigimos que el SRS esté aprobado antes de codificar. Todo eso ocurre antes de que exista una sola línea funcional. El control ocurre en semanas 9–11: Diana ejecuta Cypress, corre OWASP ZAP y coordina las sesiones UAT. Ahí ya no se previene, se mide y se documenta.
Pregunta 3: Consecuencias de cambios no controlados (Scope Creep)
El riesgo RP-02 (Scope Creep, nivel ALTO) describe exactamente esto. El escenario típico: el cliente ve el prototipo en Semana 6 y pide que el sistema maneje también el inventario de medicamentos. Si ese cambio entra sin aprobación, un desarrollador trabaja en algo no presupuestado, el módulo de expedientes se retrasa, y en Semana 10 no hay tiempo para hacer UAT con profundidad. Además, el SRS deja de reflejar lo que está en el código: la documentación y el sistema se desalinean. La solución no es rechazar todo: es canalizar. Cualquier solicitud nueva pasa por el registro de cambios, se analiza el impacto y se decide si entra, se posterga para v2.0, o se rechaza con argumento documentado.
Conclusiones

Cierre y Lecciones del Proyecto

Reflexiones finales del Grupo 7 sobre la planificación del Software de Automatización de Citas Médicas.

Conclusión 1 — Viabilidad Técnica y Cultura de Calidad

La planificación del Software de Automatización de Citas Médicas demuestra que es técnicamente viable desarrollar una solución robusta en 12 semanas, siempre que el equipo mantenga las dependencias entre fases bajo control y cumpla con los hitos del cronograma. Al desarrollar este plan entendimos que la calidad no es una revisión final, sino un hábito presente desde el primer día: desde la redacción de requisitos, la revisión constante del código entre compañeros y la definición de criterios de aceptación claros. En un sistema que maneja expedientes médicos y citas por WhatsApp, un error no es solo un fallo técnico, sino un riesgo real para la atención de un paciente.

Conclusión 2 — Sostenibilidad Financiera y Control de Configuración

Desde el punto de vista financiero, el proyecto presenta una justificación sólida con un VPN positivo de L 332,177 y una TIR del 393% que supera ampliamente la tasa de descuento del 10% anual, con un período de recuperación estimado en menos de 7 meses. Sin embargo, esa sostenibilidad a largo plazo depende de un buen control de configuración: no basta con que el software funcione hoy, debe ser mantenible cuando Meta actualice su API o cuando las clínicas requieran nuevas funciones. En el contexto de Honduras, donde la digitalización de la salud está creciendo, gestionar correctamente cada cambio es lo que convierte la tecnología en una solución real para el sector médico, y no en un problema de gestión adicional.
